# Study Desk 总功能说明

## 1. 产品定位
`Study Desk` 不是一个普通的待办事项列表，而是一个本地优先的学习 / 任务工作台。

它的核心目标不是只记录“今天要做什么”，而是把三类时间自然连起来：

- 今天正在推进的任务
- 昨天已经留下的进度与线索
- 未来某一天需要继续处理或提醒自己的安排

从产品角色来看，`Study Desk` 更像是一个“任务桌面”：

- 用户可以像摆放便签一样组织任务
- 可以边做边记录灵感、问题和对话
- 可以把任务延续到未来日期
- 可以借助 LLM 做轻量辅助
- 可以通过历史与月历视图回看自己的推进轨迹

它强调的是：

- 本地优先
- 轻量记录
- 延续推进
- 节奏调节

因此，`Study Desk` 既不是纯笔记工具，也不是单纯提醒工具，而是一个帮助用户持续推进学习和项目任务的工作台。

## 2. 当前已实现功能

### 2.1 本地服务与数据存储
系统通过本地 Python 服务 `serve_app.py` 提供页面访问、本地数据读写和 LLM 代理能力。

当前核心数据默认写入：

- `D:\projects\daily_planning\data\tasks.json`
- `D:\projects\daily_planning\data\notes.json`
- `D:\projects\daily_planning\data\messages.json`
- `D:\projects\daily_planning\data\records.json`
- `D:\projects\daily_planning\data\memory_entries.json`

这意味着：

- 任务与记录优先保存在本机
- 不依赖云端账号体系
- 用户可以直接在 D 盘管理和备份自己的数据

### 2.2 今日自由画布
“今日画布”是当前系统的主工作区。

它解决的问题是：让用户把当天真正要推进的任务直接摆在眼前，而不是埋在列表里。

在这个视图中，用户可以：

- 查看当天任务概览
- 在画布上摆放任务卡片
- 直观看到当天的活跃任务和完成情况
- 打开任务详情继续推进

### 2.3 新建学习卡片
用户可以在今日画布中创建新的学习任务卡片。

每张卡片代表一个独立任务，通常对应：

- 一个学习目标
- 一个项目步骤
- 一次需要推进的工作块

系统创建卡片后，会直接进入可编辑状态，降低用户开始记录的成本。

### 2.4 卡片操作：拖拽、缩放、颜色、归档、删除
当前任务卡片支持以下桌面式操作：

- 拖拽位置
- 自由缩放
- 切换卡片颜色
- 归档
- 删除

这些能力解决的问题分别是：

- 拖拽：按用户自己的认知顺序摆放任务
- 缩放：重要内容可以留更大空间
- 颜色：帮助用户快速识别不同任务
- 归档：把今天已处理或暂时收起的任务移出主画布
- 删除：彻底移除不再需要的卡片

### 2.5 卡片详情抽屉
点击卡片后，可以打开右侧详情抽屉。

这个抽屉是任务的主要编辑区，承载了任务的细化信息，包括：

- 标题
- 状态
- 优先级控制
- Notes
- LLM 对话
- 任务延续区

它解决的问题是：把“卡片表面保持简洁”和“任务内部信息足够完整”这两件事分开处理。

### 2.6 标题输入优化
当前标题输入已经做过一轮体验优化，重点解决中文输入法和频繁保存的冲突。

现有输入保存方式包括：

- `blur`
- `Enter`
- `debounce`
- 中文输入法 `composition` 处理

这部分解决的问题是：

- 输入过程不再被频繁打断
- 中文输入法不容易丢字
- 用户可以更自然地覆盖默认标题

### 2.7 任务状态
每张卡片支持三个状态：

- `待开始`
- `进行中`
- `已完成`

这三个状态是任务推进的基本骨架，用来表示：

- 还没启动
- 正在推进
- 已经完成

状态不仅影响卡片本身，也会影响历史记录、总结和未来的月历展示逻辑。

### 2.8 重要 / 紧急双四档滑动条
当前优先级系统已经从旧四象限升级为两条独立控制：

- `重要`
- `紧急`

每条都有四档：

- `25`
- `50`
- `75`
- `100`

这样设计的目的，是让任务优先级不再只有离散标签，而是更贴近用户对任务强弱程度的直觉判断。

### 2.9 自动分数与自动高优先级
系统当前采用自动优先级分数：

- `autoPriorityScore = importance * 0.5 + urgency * 0.5`

同时定义：

- 当 `importance >= 75` 且 `urgency >= 75` 时
- 该任务属于“重要且紧急”的自动高优先级任务

这个机制解决的问题是：

- 不需要用户每次手工调整所有卡片顺序
- 系统可以根据优先级自动帮助用户把更关键的任务往前提

### 2.10 右键排序：置顶 / 置底 / 恢复默认排序
除了自动优先级，卡片还支持右键排序操作：

- `置顶`
- `置底`
- `恢复默认排序`

这部分是给用户保留手动干预空间：

- 当系统自动排序不符合直觉时，可以直接手动压到最前或最底
- 如果之后不想继续手动控制，可以恢复默认排序

### 2.11 灵感 Notes
每张任务卡都可以记录 `Notes`。

这部分解决的是“边做边想”的问题。  
用户在推进任务时，往往会出现：

- 零碎灵感
- 临时提醒
- 分步骤想法
- 下一步抓手

Notes 让这些信息不用等到整理完再写，而是可以先存下来。

### 2.12 任务内 LLM 对话
每张任务卡都支持与 LLM 的独立对话。

它的用途不是做一个泛化聊天窗口，而是围绕当前任务提供辅助，例如：

- 询问下一步怎么推进
- 讨论任务拆解方式
- 记录问题和推理线索

这样每张任务卡不仅有“做什么”，还有“我是怎么想的”。

### 2.13 每日总结
系统支持对当天任务进行总结。

总结会综合：

- 任务状态
- Notes
- 任务内对话
- 相关延续信息

它解决的问题是：

- 帮助用户在一天结束时形成一个回顾
- 把分散在卡片里的信息收束成一个整体视角

### 2.14 历史记录
系统提供按天查看的历史记录。

在历史记录中，用户可以回看某一天的：

- 任务
- 状态
- 灵感
- 对话
- 总结
- 延续信息

这个模块的意义是：让每天的推进不是消失的，而是可追溯、可回看的。

### 2.15 任务延续区
“任务延续区”是当前系统非常关键的一个模块，它替代了原本更偏“结项证明”的 memory 设计。

当前任务延续区包括：

- `上次进度`
- `相关路径`
- `相关文件`
- `DDL`

它解决的问题是：

- 当任务没有在一天内做完时，第二天如何无缝接上
- 当任务隔几天再打开时，如何快速找回上下文

相比“完成定义”，它更强调“继续推进”。

### 2.16 Quote 一句日历
系统已经引入本地 Quote 机制，数据来源于 `quotes.json`。

当前 Quote 分为三类：

- `start`
- `push`
- `random`

它不是提醒工具，而是一个轻量的节奏调节器，用来在用户开始或延续任务时，给出一句不打扰但有温度的提示。

### 2.17 每天首次打开的今日 Quote
当前系统在每天首次打开页面时，会在日期下方展示一条当日 Quote。

它的作用是：

- 降低开始一天任务时的心理阻力
- 给当天工作一个轻量启动感
- 不打扰操作，不形成强提醒

### 2.18 Copy To Tomorrow 弹层
当前系统已经把原来的直接复制操作升级为弹层式流程，用于把任务延续到下一天。

当前弹层支持：

- Notes 多选
- 摘要模式选择
- 可选附加一句日历

它解决的问题是：

- 用户不需要把整张卡片原封不动搬过去
- 可以只带走真正值得延续的内容

## 3. 4.27 新增 / 迭代功能

这一部分是基于当前产品方向进一步明确和扩展的功能说明。  
它们代表产品的下一阶段重点，不等同于已经全部完成上线。

### 3.1 “复制到明天”升级为“安排到日期”
原本的“复制到明天”只支持把任务延续到次日。  
4.27 的迭代方向，是把它升级为更通用的：

- `安排到日期`

这意味着：

- 用户可以把任务安排到任意目标日期
- 这个能力不再只是“继续明天做”
- 也可以作为未来提醒、未来排程、某天必须完成的事情

目标日期将使用与历史记录一致的日期选择器，保证交互统一。

新任务生成后需要带来源标记，例如：

- `↪ from 2026/04/27`

这能帮助用户区分：

- 当天新建任务
- 从过去某一天安排过来的任务

### 3.2 安排到日期弹层内容
新的“安排到日期”弹层将支持更细粒度的内容选择。

用户可以决定带过去哪些信息，包括：

- 任务标题
- 勾选任意 `Notes`
- 任务延续区中的不同字段
  - 上次进度
  - 相关路径
  - DDL
- 当前 LLM 对话摘要
- 下一步计划
- 一句日历

这个迭代解决的问题是：

- 未来任务不再只是“标题复制”
- 而是可以带着足够的上下文一起过去
- 同时又不会把所有内容无差别复制，避免信息过载

### 3.3 当前 LLM 对话摘要
在 4.27 的定版里，“当前 LLM 对话摘要”明确采用：

- 用户勾选后自动生成摘要

而不是：

- 复制完整对话
- 或让用户手工整理对话

这样做的目的，是让未来任务保留对话价值，但不把任务卡变成一长段聊天记录。

输出要求也被明确限制为：

- 自然语言
- 不出现 `##`
- 不出现 Markdown 标题
- 不出现 JSON
- 不出现代码块

### 3.4 一句日历在“安排到日期”中的作用
一句日历仍然保留为一个可选项。

如果用户在“安排到日期”中勾选它，那么它会被附加到未来任务内容中，作为一种轻量的情绪与节奏连接。

格式要求也已经明确：

- 必须换行展示
- 不能挤成摘要中的一句尾巴

它在这个场景中扮演的角色不是任务说明，而是“未来某天重新打开这张卡时的轻提示”。

### 3.5 月历模式
在“今日画布”和“历史记录”之外，系统计划新增：

- `月历模式`

月历模式的目的，是让用户从“按天推进”提升到“按月观察”。

它解决的问题包括：

- 哪些任务已经被安排到未来
- 哪些天积压了更多任务
- 哪些天已经完成较多
- 哪些任务与 DDL 或重点日程相关

月历按自然月展示，每个日期格子最多显示两条任务，超出部分用 `+n` 表示。

### 3.6 月历中的任务标记与状态
月历里的任务会有更轻量但清晰的状态标记：

- `↪`：表示这张任务是从过去某天安排过来的
- `★`：表示这张任务重要且紧急
- `✓`：表示该任务已完成，且在视觉上变灰
- `+n`：表示这个日期还有更多未展开任务

这些标记能让用户在一个月视角下快速识别：

- 重点任务
- 延续任务
- 已完成任务
- 信息密度过高的日期

### 3.7 月历任务展示顺序
当前 4.27 版本中，月历任务展示顺序暂定为：

1. 安排过来的任务
2. 重要且紧急的任务
3. 已完成任务
4. 该日新增普通任务

这是当前阶段的排序约定，用来突出“未来安排”和“重点任务”。  
其中“已完成任务”的位置已经记录为一个可后续观察的点，未来可以根据真实使用体验再决定是否继续后移。

### 3.8 月历交互
月历模式的交互已经明确为：

- 点击日期格子
  - 打开该天的侧栏
  - 侧栏中展示该天的历史记录概览
- 点击格子中的任务名
  - 直接打开该任务详情
- 鼠标悬停 `+n`
  - 显示剩余任务的 tooltip 预览
- `+n`
  - 当前不作为跳转入口

这套交互的目标是：

- 月历保持轻量
- 细节通过侧栏和 tooltip 补充
- 不让一个月视图变成信息爆炸的“全量列表”

## 4. 关键用户流程

### 4.1 当天开始任务
用户打开 `Study Desk` 后：

1. 看到当天日期与一句日历
2. 进入今日画布
3. 新建或打开已有任务卡片
4. 通过任务状态、Notes、LLM 对话和任务延续区开始推进

这个流程解决的是“如何低成本进入工作状态”。

### 4.2 在任务进行中记录上下文
用户在推进过程中可以：

1. 写 Notes
2. 向 LLM 提问
3. 更新任务状态
4. 把当前进展、路径和 DDL 写入任务延续区

这个流程解决的是“不要等想清楚再记录，而是边做边沉淀”。

### 4.3 一天结束时做收尾
用户在一天结束时可以：

1. 生成每日总结
2. 标记已完成任务
3. 归档不再需要放在画布上的卡片
4. 决定哪些任务需要延续到未来

这个流程解决的是“把今天收住，而不是让任务在明天重新变乱”。

### 4.4 把任务延续到未来
当用户觉得某个任务未来某天还要继续推进时，可以：

1. 打开“安排到日期”
2. 选择目标日期
3. 勾选 Notes、任务延续区字段、LLM 对话摘要、下一步计划、一句日历等内容
4. 生成未来任务卡片

这个流程解决的是“如何让未来某一天打开任务时，不需要重新回忆上下文”。

### 4.5 回看过去与查看未来排程
用户可以通过：

- 历史记录：回看某一天
- 月历模式：总览一个月

来理解：

- 自己已经做过什么
- 哪些任务已经完成
- 哪些任务被安排到了未来

## 5. 数据对象概览

### 5.1 任务卡片
任务卡片是系统的核心对象。

它承载的信息包括：

- 日期
- 标题
- 状态
- 位置与尺寸
- 颜色
- 重要 / 紧急值
- 排序状态
- 是否归档
- 是否来自过去某天安排

### 5.2 Notes
Notes 是与任务绑定的灵感记录。

它们通常用来保存：

- 碎片想法
- 关键提醒
- 下一步提示

### 5.3 LLM 对话
LLM 对话也是和任务绑定的对象。

它们用于保存：

- 用户就当前任务提出的问题
- LLM 返回的思路或建议

### 5.4 每日总结
每日总结是按日期保存的日级记录。

它帮助用户把一天内分散的信息收束成一份整体回顾。

### 5.5 任务延续区
任务延续区是帮助未来继续推进任务的上下文对象。

它保存的是：

- 上次进度
- 路径
- 文件
- DDL

### 5.6 Quote
Quote 是一句日历的数据对象。

它按类型分为：

- `start`
- `push`
- `random`

主要用于：

- 每天首次打开时的轻提示
- 安排到日期时的情绪连接

## 6. 当前仍需注意的问题

### 6.1 置顶 / 自动优先级冲突
当前系统已经有自动优先级和手动置顶 / 置底，但两者之间的语义关系还没有完全稳定。

尤其是：

- 手动置顶是否应该绝对覆盖自动排序
- 当用户后续再次调整重要 / 紧急时，是否应该打破先前手动置顶的结果

这个问题已经在项目记录中明确保留，后续仍需继续打磨。

### 6.2 月历模式仍属于迭代设计重点
月历模式的目标、标记、排序与交互已经有比较明确的产品定义，但它本身仍处于 4.27 的重点迭代范围中。

因此，需要特别区分：

- 现有已实现主流程
- 与正在补充设计、准备继续推进的月历能力

### 6.3 未来任务的信息密度控制
随着“安排到日期”能力增强，未来某天的任务卡可能会带上：

- Notes
- 延续区字段
- LLM 摘要
- 一句日历

这虽然提升了上下文完整度，但也带来新的问题：

- 信息会不会太重
- 哪些内容默认该展示
- 哪些内容应该折叠

这是后续仍需要持续观察和收口的点。

## 7. 后续扩展方向

### 7.1 月历模式完善
后续可以继续完善：

- 月历侧栏内容
- tooltip 细节
- 日期格子的视觉压缩方案
- future task / DDL 的更强高亮方式

### 7.2 安排到日期的细化
后续可以继续讨论：

- 是否支持选择更多任务延续区字段
- 是否支持更细粒度的“带过去内容模板”
- 是否支持批量安排多个任务

### 7.3 任务延续与未来排程联动
未来可以进一步加强“延续区”和“安排到日期”的联动，让用户不只是复制信息，而是形成真正的推进链条。

### 7.4 桌面化与悬浮化形态
从长期来看，`Study Desk` 还可以从网页/PWA 继续扩展到：

- 更像桌面便签的形态
- 悬浮聊天框
- 更低启动成本的一键打开体验

### 7.5 Quote 与节奏系统扩展
一句日历未来还可以继续扩展：

- 本地 quote 库管理
- 不同情境下的 quote 分发策略
- 更成熟的节奏调节机制

---

`Study Desk` 的整体方向，不是把任务越堆越多，而是让“今天、昨天、未来某天”之间的连接越来越自然。
